# Roadmap вАФ Anaj Bahi (а§Еа§®а§Ња§Ь а§ђа§єа•А)

> **Product name:** **Anaj Bahi** (а§Еа§®а§Ња§Ь а§ђа§єа•А) вАФ literally "Grain Ledger" (*anaj* = grain, *bahi* = the traditional Indian ledger/account book, *bahi-khata*). It is bilingual-friendly, instantly meaningful to the user, and reads naturally in both Devanagari and Latin script. Kept as the final name.

---

## What This App Does

Anaj Bahi is an installable, offline-first mobile PWA that replaces a village grain trader's paper receipt-and-payment book. The trader buys grain from farmers, and for each purchase records the farmer, the grain type(s), the price per quintal, and every individual sack weight вАФ sack by sack вАФ then the app computes deductions, net weight, per-grain amounts, and the bill total exactly. Bills are saved locally on the phone (IndexedDB) with zero internet, browsable in a searchable list, and (in later phases) tracked for partial payments and outstanding balances, shared as an image receipt over WhatsApp, and backed up to the cloud when signal returns.

## Who Uses It

A single user: a grain trader in a small Indian village (the product owner's father). Low-end Android phone, intermittent/absent connectivity, one-handed use in a busy mandi (market), comfortable in Hindi (Devanagari) with some English. Not a power user вАФ big buttons, few taps, no jargon.

## Core Problem Being Solved

Today the trader keeps purchases and payments in a paper ledger: slow to write, error-prone arithmetic (sack weights, deductions, per-quintal pricing), hard to search ("what did I buy from Ramesh last week?"), easy to lose, and impossible to share a clean receipt. Anaj Bahi digitises exactly this workflow вАФ fast sack-by-sack capture, exact money math, instant search, and a shareable receipt вАФ without requiring internet.

## Success Criteria

- [ ] Trader can create a complete purchase bill (farmer + вЙ•1 grain line + sack weights + deductions) and save it with the phone in airplane mode; reopening the app shows the saved bill unchanged.
- [ ] Sack-by-sack entry lets him add вЙ•30 sack weights one-handed, seeing the running list, running count, and running total above the input the whole time.
- [ ] Line net weight, line amount, and bill total are computed exactly (unit-correct: quintal = 100 kg) and match hand calculation on the documented worked examples, including multiple deductions per line.
- [ ] The Hindi/English toggle switches all on-screen chrome and labels immediately, with the preference remembered across launches.
- [ ] The app installs to the Android home screen and launches full-screen offline (service worker + manifest).
- [ ] Every later-phase feature (payments, due list, cloud sync, share, search filters) appears as a clearly-labelled "coming soon" stub in Phase 1 вАФ never mistaken for a bug.

## What This App Does NOT Do (Out of Scope)

- **No AI / LLM / agent of any kind.** This is a deterministic local-first data/ledger app. There is no model, no prompt, no `spec/agent.md` graph. **The Firebase multi-tenant redesign (Phases 6вАУ9) does not change this** вАФ Firebase Auth + Firestore are ordinary client SDKs, still no AI.
- ~~No multi-user, roles, or accounts.~~ **Multi-user is now IN scope (Phases 6вАУ9):** many independent **businesses**, each with **owners** and **employees** sharing one ledger, via Firebase Auth (**Google sign-in**) + Cloud Firestore. (Phases 1вАУ5 shipped single-user; the Phase-4 FastAPI+SQLite personal-backup is **superseded** by Firestore.) Still out of scope: cross-business sharing, org hierarchies beyond owner/employee, and promoting an employee to owner.
- No dashboard, charts, or analytics in early phases вАФ only the searchable bill list.
- No phone/push notifications вАФ the "due soon / overdue" list is in-app only.
- No selling/inventory/GST/tax invoicing вАФ this records *purchases from farmers*, not sales.
- No payment gateway or actual money movement вАФ payments are ledger records only.
- No web/desktop-optimised layout вАФ mobile-first, portrait, Android-first.

## Key Constraints

- Must run and persist **fully offline**; Phase 1 has **no backend, no server DB, no API keys, no `.env`**.
- Must be usable on a low-end Android phone: small bundle, static export, big touch targets, no heavy runtime.
- Money and weight math must be **exact and unit-correct** and isolated in a pure, unit-tested module.
- Bilingual Hindi (Devanagari) + English throughout, including the shared receipt (later phase).
- Currency вВє INR; weights in kg; price per **quintal (100 kg)**.

### Phase 7 вАФ Firestore shared store, offline persistence, migration; remove old backend

- **Goal:** The ledger moves to **Cloud Firestore** with **IndexedDB offline persistence as the store** вАФ bills/farmers/grain types/profile are **shared, business-scoped, live across members and devices**; offline writes queue locally and auto-sync on reconnect. The trader's existing **local data migrates once** into their business. Settings gains an **online/sync-status + "Sync now"** indicator. The old **FastAPI backend, `lib/sync/*`, and `SyncSettings` are removed**.
- **Capabilities:** [firestore-store](capabilities/firestore-store.md) (re-points `lib/db/repo`, adds offline-persistent reads, one-time migration, sync-status/"Sync now", and removes the retired backend). _This phase is deliberately a single deep data-layer capability with several parallel slices; roles/attribution/activity follow in Phases 8вАУ9._
- **Independent slices (parallel build units) вАФ disjoint file paths:**
  - `slice-a` (**repo re-point + reactive hooks**). **Deps: Phase-6 auth (active business) contract.** Rewrite `frontend/src/lib/db/repo.ts` to be Firestore-backed + business-scoped (same signatures) + `setActiveBusiness`; add Firestore `onSnapshot` hooks (`useBills`/`useBill`/`useFarmers`/`useGrainTypes`); wire `AuthProvider` to call `setActiveBusiness`. **+ unit/integration tests against the Firestore emulator or a real test project.**

### Phase 8 вАФ Attribution, employee management, personal/business profiles, Security Rules

- **Capabilities:** [employee-management](capabilities/employee-management.md), [personal-profile](capabilities/personal-profile.md), and **per-action attribution** (createdBy snapshots on bills/payments вАФ see [firestore-store](capabilities/firestore-store.md) / [data.md](data.md)).
